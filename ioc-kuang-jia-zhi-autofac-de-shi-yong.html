AutoFac依赖注入框架的使用

AutoFac的基本流程：

声明容器->注册实例->解析对象->调用方法

Demo类创建

using System;

namespace DemoApp
{

  public interface IOutput
  {
    void Write(string content);
  }

  //继承接口IOutput
  public class ConsoleOutput : IOutput
  {
    public void Write(string content)
    {
      Console.WriteLine(content);
    }
  }

  public interface IDateWriter
  {
    void WriteDate();
  }

  //继承接口IDateWriter
  public class TodayWriter : IDateWriter
  {
    private IOutput _output;
    //演示构造函数注入
    public TodayWriter(IOutput output)
    {
      this._output = output;
    }

    public void WriteDate()
    {
      //若没有使用IoC容器此处报错
      this._output.Write(DateTime.Today.ToShortDateString());
    }
  }
}

1.创建生成器 Builder

创建注册组件/服务的生成器。 

var builder = new ContainerBuilder();

2.注册组件

2.1 通过类型注册

var builder = new ContainerBuilder();
builder.RegisterType<ConsoleLogger>().As<ILogger>();
builder.RegisterType(typeof(ConfigReader)).As<ILogger>();

2.1.1 构造函数匹配规则

  • 当使用基于反射的组件时，AutoFac自动为你的类从容器中寻找匹配拥有最多参数的构造方法。

例如，此时拥有三个构造函数的类：

public class MyComponent
{
    public MyComponent() { /* ... */ }
    public MyComponent(ILogger logger) { /* ... */ }
    public MyComponent(ILogger logger, IConfigReader reader) { /* ... */ }
}

现在我们在容器中注册组件和服务：

var builder = new ContainerBuilder();
builder.RegisterType<MyComponent>();
builder.RegisterType<ConsoleLogger>().As<ILogger>();
var container = builder.Build();

using(var scope = container.BeginLifetimeScope())
{
  var component = scope.Resolve<MyComponent>();
}

当解析组件时，Autofac发现ILogger已经被注册（第三行），但是我们并没有注册IConfigReader，所以它寻找匹配最多参数的构造方法，即第二个构造函数。

2.1.2 指定构造函数

我们可以使用UsingConstructor手动指定一个构造函数。

builder.RegisterType<MyComponent>()
       .UsingConstructor(typeof(ILogger), typeof(IConfigReader));

要注意的是, 在解析时你仍然需要提供必要的参数，否则在你尝试解析对象时将出现错误.

你可以 在注册时传参 或 在解析时传参.

2.2 实例组件

通过使用 RegisterInstance 方法 ：

var output = new StringWriter();
builder.RegisterInstance(output).As<TextWriter>();

2.2.1 自己控制组件的生命周期

当使用new实例化一个组件之时，必然会遇到内存释放问题，Autofac会自动处理已注册组件的释放，当时也可以自己控制生命周期。

我们可以使用ExternallyOwned 方法来注册实例：

var output = new StringWriter();
builder.RegisterInstance(output)
       .As<TextWriter>()
       .ExternallyOwned();

2.3 Lambda表达式组件

Autofac接收一个委托或者lambda表达式, 用作组件创建者: 

builder.Register(c => new A(c.Resolve<B>()));

2.3.1 参数注入

builder.Register(c => new A(){ MyB = c.ResolveOptional<B>() });

2.4 暴露服务

我们注册组件时，需要告诉Autofac，组件暴露了哪些服务。

使用类型注册(RegisterType)时，大多数情况下组件自身将被暴露。

// "CallLogger"将被暴露
builder.RegisterType<CallLogger>();

组件能够被它暴露的服务解析：

//可以解析成功
scope.Resolve<CallLogger>();
//不能解析成功，因为ILogger未能暴露
scope.Resolve<ILogger>();

我们可以让一个组件暴露一个或多个服务 ：使用 As

builder.RegisterType<CallLogger>()
       .As<ILogger>() //暴露ILogger服务
       .As<ICallInterceptor>(); //暴露ICallInterceptor服务

暴露服务之后，你就可以解析基于该服务的组件了。

不过要注意的是，一旦你将组件暴露为一个特定的服务，那么默认的服务/组件类型将被覆盖：

//解析成功
scope.Resolve<ILogger>();
scope.Resolve<ICallInterceptor>();
//解析失败
scope.Resolve<CallLogger>();

使用AsSelf方法可以让组件技能暴露一系列特定服务，也能让它暴露默认服务。

builder.RegisterType<CallLogger>()
       .AsSelf() //AsSelf
       .As<ILogger>()
       .As<ICallInterceptor>();

2.5 默认注册

如果多个组件暴露了相同的服务，那么Autofac将会选取最后注册的组件作为服务提供方：

namespace DemoIBLL
{
    //接口IUserBLL
    public interface IUserBLL
    {
        string GetUserInformation();
    }

    //UserBLL
    public class UserBLL : IUserBLL
    {
         public string GetUserInformation()
         {
             return "我是wk。";
         }
    }

    //SuperUserBLL
    public class SuperUserBLL : IUserBLL
    {
        public string GetUserInformation()
        {
            return "我是超人";
        }
    }
}
ContainerBuilder builder = new ContainerBuilder();
builder.RegisterType<UserBLL>().As<IUserBLL>();
//这个注册的组件将会成为服务的提供商
builder.RegisterType<SuperUserBLL>().As<IUserBLL>(); 
IContainer container = builder.Build();
IUserBLL userBLL = container.Resolve<IUserBLL>();
var result = userBLL.GetUserInformation();
Console.WriteLine(result); //输出：我是超人

2.6 两种注册扩展方式

  • OnlyIf() - 提供一个表达式, 使用一个 IComponentRegistry 来决定注册是否发生.（仅当XX服务注册时注册）
  • IfNotRegistered() - 有其他服务已被注册的情况下阻止注册发生的快捷方法. （XX服务注册时不注册）

这些方法在 ContainerBuilder.Build() 时执行并且以实际组件注册的顺序执行.

例子：

var builder = new ContainerBuilder();

//只注册ServiceA
builder.RegisterType<ServiceA>()
       .As<IService>();
builder.RegisterType<ServiceB>()
       .As<IService>()
       //如果IService服务被注册，则取消注册
       .IfNotRegistered(typeof(IService)); 

//顺序：A先被注册，因为它在HandleB注册之前运行
//C将不会被注册，因为它运行在HandlerB之后
builder.RegisterType<HandlerA>()
       .AsSelf()
       .As<IHandler>()
       .IfNotRegistered(typeof(HandlerB));
builder.RegisterType<HandlerB>()
       .AsSelf()
       .As<IHandler>();
builder.RegisterType<HandlerC>()
       .AsSelf()
       .As<IHandler>()
       .IfNotRegistered(typeof(HandlerB));


//Manager将被注册，因为IService和HandlerB已经注册
builder.RegisterType<Manager>()
       .As<IManager>()
       .OnlyIf(reg =>
         reg.IsRegistered(new TypedService(typeof(IService))) &&
         reg.IsRegistered(new TypedService(typeof(HandlerB))));

2.7 注册时传参

我们在注册组件的时候，可以提供一组参数，用于该组件在服务解析的时候使用，当然也可以在服务解析的时候提供参数。

2.7.1 可使用的参数类型

Autofac提供了多种不同的参数匹配机制：

  • NamedParameter - 通过名字匹配目标参数

  • TypedParameter - 通过类型匹配目标参数 (需要匹配具体的类型)

  • ResolvedParameter - 复杂参数的匹配

    NamedParameter 和 TypedParameter 只支持常量值.

    ResolvedParameter 可以用于提供不同的值来从容器中动态获取对象, 例如, 通过名字解析服务.

2.7.2 反射组件的参数

当年注册一个基于反射的组件时，类型的构造方法也许会需要一个无法从容器中解析出来的参数，例如常量类型的参数，你可以在注册时提供该参数值。

例如：构造函数ConfigReader中需要传入一个configSectionName

public class ConfigReader : IConfigReader
{
  public ConfigReader(string configSectionName)
  {
    // 存储配置节点名configSectionName
  }

  // 读取基于该节点的配置
}

你可以使用lambda表达式组件: 

builder.Register(c => new ConfigReader("sectionName")).As<IConfigReader>()

或者在反射组件注册时传参: 

//使用NAMED parameter
builder.RegisterType<ConfigReader>()
       .As<IConfigReader>()
       .WithParameter("configSectionName", "sectionName");


// 使用TYPED parameter:
builder.RegisterType<ConfigReader>()
       .As<IConfigReader>()
       .WithParameter(new TypedParameter(typeof(string), "sectionName"));

// 使用RESOLVED parameter:
builder.RegisterType<ConfigReader>()
       .As<IConfigReader>()
       .WithParameter(
         new ResolvedParameter(
           (pi, ctx) => pi.ParameterType == typeof(string) && pi.Name == "configSectionName",
           (pi, ctx) => "sectionName"));

2.7.3 Lambda表达式组件的参数

使用lambda表达式组件注册, 你可以不用 在注册时 传入参数值, 而是可以 在服务解析时 传入具体的参数值 。

/*
使用两个参数注册委托:
c =动态解析依赖项的当前IComponentContext
p =带有传入参数集的IEnumerable<Parameter>
*/
builder.Register((c, p) =>
                 new ConfigReader(p.Named<string>("configSectionName")))
       .As<IConfigReader>();

//当解析时你需要传入这些值
var reader = scope.Resolve<IConfigReader>(new NamedParameter("configSectionName", "sectionName"));

2.8 属性注入&方法注入

构造函数注入是一种传值给组件的首选方法，但是同样可以使用属性和方法注入来传值。

2.8.1 属性注入

如果组件是一个Lambda表达式，我们可以使用对象构造器：

builder.Register(c => new A { B = c.Resolve<B>() });

如果组件是一个反射组件，我们可以使用 PropertiesAutowired() 修饰语来注入属性: 

builder.RegisterType<A>().PropertiesAutowired();

如果你需要绑定一个特定的属性和它的值, 使用 WithProperty() 修饰语: 

builder.RegisterType<A>().WithProperty("PropertyName", propertyValue);

2.8.2 方法注入

想要调用一个方法来设置组件上的某个值, 最简单的方法是使用 lambda表达式组件 然后在activator中进行正确的方法调用:

builder.Register(c => {
  var result = new MyObjectType();
  var dep = c.Resolve<TheDependency>(); 
  result.SetTheDependency(dep); //SetTheDependency为方法注入口
  return result;
});

如果不使用注册Lambda表达式，你可以添加一个激活时事件处理程序 (activating event handler): 

builder
  .Register<MyObjectType>()
   //OnActivating 激活时事件处理程序
  .OnActivating(e => {
    var dep = e.Context.Resolve<TheDependency>();
    e.Instance.SetTheDependency(dep);
  });

2.9 扫描程序集

Autofac可以在程序集中通过约定找到和注册组件.

2.9.1 扫描类型

//获得当前代码运行的程序集
var dataAccess = Assembly.GetExecutingAssembly();

builder.RegisterAssemblyTypes(dataAccess)
        //默认程序集上的类都会被注册，使用linq表达式过滤注册类型
       .Where(t => t.Name.EndsWith("Repository"))
       .AsImplementedInterfaces();
2.9.1.1 过滤类型

在版本4.8以上，使用 PublicOnly()扩展方法，可以只让公有方法被注册。

builder.RegisterAssemblyTypes(asm)
       .PublicOnly();

使用where()表达式来过滤注册类型：

builder.RegisterAssemblyTypes(asm)
        //只注册以"repository"为结尾的类
       .Where(t => t.Name.EndsWith("Repository"));

使用Except()表达式来排除类型：

builder.RegisterAssemblyTypes(asm)
       .Except<MyUnwantedType>();

Except()还可以自定义被排除类型的规则：

builder.RegisterAssemblyTypes(asm)
       .Except<MyCustomisedType>(ct =>
          ct.As<ISpecial>().SingleInstance());

可以使用多个过滤条件：

builder.RegisterAssemblyTypes(asm)
       .PublicOnly()
       .Where(t => t.Name.EndsWith("Repository"))
       .Except<MyUnwantedRepository>();
2.9.1.2 指定服务

As()指定单一服务： 

builder.RegisterAssemblyTypes(asm)
       .Where(t => t.Name.EndsWith("Repository"))
        //指定注册IRepository服务，若一个类使用了其它服务，将报错未注册
       .As<IRepository>();

AsImplementedInterfaces() 把一个类注册给它实现的全部接口。

AsSelf()注册自身。

2.9.2 扫描模块

我们通过 RegisterAssemblyModules() 方法进行模块扫描。

假设下面两个普通的模块类在同一个程序集中, 并且每个模块注册一个组件: 

public class AModule : Module
{
  protected override void Load(ContainerBuilder builder)
  {
    builder.Register(c => new AComponent()).As<AComponent>();
  }
}

public class BModule : Module
{
  protected override void Load(ContainerBuilder builder)
  {
    builder.Register(c => new BComponent()).As<BComponent>();
  }
}

RegisterAssemblyModules()的重载，不接受类型参数，他将会注册所提供程序集中的所有IModule类，例如在下面的示例中，所有的模块都将被注册：

var assembly = typeof(AComponent).Assembly;
var builder = new ContainerBuilder();

// Registers both modules
builder.RegisterAssemblyModules(assembly);

使用泛型参数的 RegisterAssemblyModules()的重载允许你指定被注册模块：

var assembly = typeof(AComponent).Assembly;
var builder = new ContainerBuilder();

// Registers AModule but not BModule
builder.RegisterAssemblyModules<AModule>(assembly);

3. 解析服务

我们在注册完并且暴露相应的服务之后，你可以从创建的容器或者其子生命周期中接续服务。

使用Resolve()方法实现：

var builder = new ContainerBuilder();
builder.RegisterType<MyComponent>().As<IService>();
var container = builder.Build();
//从子容器生命周期中解析服务，并不是直接从容器中
using(var scope = container.BeginLifetimeScope())
{
  var service = scope.Resolve<IService>();
}

有时在我们的应用中也许可以从根容器中解析组件, 然而这么做有可能会导致内存泄漏. 推荐你总是从生命周期中解析组件, 以确保服务实例被妥善地释放和垃圾回收. 在 控制作用域和生命周期章节 阅读更多相关内容.

如果你并不清楚一个服务是否被注册，那么可以通过 ResolveOptional() 或 TryResolve() 尝试解析: 

//如果服务被注册了，那么就可以获得服务，如果没有被注册，则获得null
var service = scope.ResolveOptional<IService>();

IProvider provider = null;
if(scope.TryResolve<IProvider>(out provider))
{
  // 如果注册了，则对解析后的的提供者执行某些操作
}

与2.7类似，选一种方式即可。

3.2 隐式关系类型

4. 生命周期

当你解析服务时, Autofac将会追踪被解析的可释放/可销毁 (IDisposable) 组件. 在工作单元最后, 你释放了相关的生命周期作用域然后Autofac将会自动清理/释放那些被解析的服务.

4.1开始使用生命周期作用域

4.1.1创建一个新的生命周期作用域

你可以通过在一个已存在的生命周期上调用 BeginLifetimeScope() 方法来创建另一个生命周期作用域 。

生命周期可以嵌套，并且是需要释放的，在释放之时会追踪组件的释放，所以需要加using或者手动调用Dispose()

using(var scope = container.BeginLifetimeScope())
{
  //从根容器的子范围内解析服务
  var service = scope.Resolve<IService>();
  // 嵌套
  using(var unitOfWorkScope = scope.BeginLifetimeScope())
  {
    var anotherService = unitOfWorkScope.Resolve<IOther>();
  }
}

4.1.2 向生命周期作用域中添加注册

我们可以在创建生命周期作用域的同时随手添加注册。

使用情景:

  • 需要在作用域内注册一些不想全局注册的东西
using(var scope = container.BeginLifetimeScope(
  //传递一个引用 ContainerBuilder 的lambda表达式
  builder =>
  {
    builder.RegisterType<Override>().As<IService>();
    builder.RegisterModule<MyModule>();
  }))
{
  // The additional registrations will be available
  // only in this lifetime scope.
}

4.2 实例作用域

实例作用域决定了对于相同的服务解析出的实例如何在请求之间被共享.

当一个请求的完成需要某个服务，Autofac可以返回单一实例，也可以每次返回新的实例（每个依赖的作用域），或者返回某种上下文共享的单一实例，例如每一次请求，每一个线程。

4.2.1 每次都返回不同的实例（InstancePerDependency）

在其他容器中也被称为 ‘transient’ 或者 ‘factory’ .

对于一个服务每次请求都会返回一个唯一的实例。

如果没有指定特定的选项，这个行为是默认的。

var builder = new ContainerBuilder();

builder.RegisterType<Worker>();
// 上下两种方式等同
builder.RegisterType<Worker>().InstancePerDependency();
using(var scope = container.BeginLifetimeScope())
{
  for(var i = 0; i < 100; i++)
  {
    //循环一百次就返回100个不同的实例
    var w = scope.Resolve<Worker>();
    w.DoWork();
  }
}

4.2.2 返回单一实例（SingleInstance）

相当于单例，在根容器和所有的嵌套作用域中所有请求都返回的是同一个实例。

var builder = new ContainerBuilder();
builder.RegisterType<Worker>().SingleInstance();
//从根容器中解析获得实例（不推荐）
var root = container.Resolve<Worker>();

//我们可以从嵌套的生命周期范围的任何级别、任意次数解析worker。
using(var scope1 = container.BeginLifetimeScope())
{
  for(var i = 0; i < 100; i++)
  {
    var w1 = scope1.Resolve<Worker>();
    using(var scope2 = scope1.BeginLifetimeScope())
    {
        //root w1 w2返回的都是同一实例
      var w2 = scope2.Resolve<Worker>();
    }
  }
}

4.2.3 在每个作用域中为同一实例（InstancePerLifetimeScope）

每个生命周期作用域的组件在每个嵌套的生命周期作用域中最多只会有一个单一实例.

var builder = new ContainerBuilder();
builder.RegisterType<Worker>().InstancePerLifetimeScope();
using(var scope1 = container.BeginLifetimeScope())
{
  for(var i = 0; i < 100; i++)
  {
    //每个w1都是同一个实例
    var w1 = scope1.Resolve<Worker>();
  }
}
using(var scope2 = container.BeginLifetimeScope())
{
  for(var i = 0; i < 100; i++)
  {
    //每个w2都是同一实例，但是w1和w2不是同一实例
    var w2 = scope2.Resolve<Worker>();
  }
}

4.2.4 通过匹配返回同一实例（InstancePerMatchingLifetimeScope）

类似于InstancePerLifetimeScope但是可以通过设置和传入标签更精准的控制实例的共享。

大多数的应用中, 只需一层的容器嵌套就足以表示工作单元的作用域. 如果需要更多的嵌套层级 (例如, 全局->请求->事务) 组件可以考虑使用标签来在层级关系中特定的层级共享. 

var builder = new ContainerBuilder();
builder.RegisterType<Worker>().InstancePerMatchingLifetimeScope("myrequest");

在开始一个生命周期时提供标签值来关联，如果提供的名称不匹配，将会抛出异常。

using(var scope1 = container.BeginLifetimeScope("myrequest"))
{
  for(var i = 0; i < 100; i++)
  {
    var w1 = scope1.Resolve<Worker>();
    using(var scope2 = scope1.BeginLifetimeScope())
    {
      //w1和w2为同一实例
      var w2 = scope2.Resolve<Worker>();
    }
  }
}

using(var scope3 = container.BeginLifetimeScope("myrequest"))
{
  for(var i = 0; i < 100; i++)
  {
    //w3和之前解析的w1和w2不同
    var w3 = scope3.Resolve<Worker>();
    using(var scope4 = scope3.BeginLifetimeScope())
    {
      //w3=w4<>w1&w2
      var w4 = scope4.Resolve<Worker>();
    }
  }
}


using(var noTagScope = container.BeginLifetimeScope())
{
  //会抛出异常，因为没有指定标签，在此作用域中Worker未被注册
  var fail = noTagScope.Resolve<Worker>();
}

4.2.5 每一次请求返回同一实例(Instance Per Request)

仅在 ASP.NET web forms 和 MVC 中应用. 

var builder = new ContainerBuilder();
builder.RegisterType<Worker>().InstancePerRequest();

4.2.6 每个Owned一个实例 ( InstancePerOwned)

Owned 隐式关联类型创建嵌套的生命周期范围。使用 instance-per-owned 注册，可将依赖限定在 owned 实例中。 

var builder = new ContainerBuilder();
builder.RegisterType<MessageHandler>();
builder.RegisterType<ServiceForHandler>().InstancePerOwned<MessageHandler>();

本例中 ServiceForHandler 服务会限制在 MessageHandler 实例范围内。 

    using(var scope = container.BeginLifetimeScope())
    {
      // MessageHandler 和附属的 ServiceForHandler 
      // 在 scope 下面的一个微型的 lifetime scope 中。
      // 解析 Owned<T> 需要程序员负责执行清理工作。
      Owned<MessageHandler> h1 = scope.Resolve<Owned<MessageHandler>>();
      h1.Value.Method();
      h1.Dispose();
    }

4.2.7 线程作用域（ThreadScope）

使用InstancePerLifetimeScope，一个线程为一个生命周期

var builder = new ContainerBuilder();
builder.RegisterType<MyThreadScopedComponent>()
       .InstancePerLifetimeScope();
var container = builder.Build();

这样, 每个线程就有了它各自的生命周期作用域: 

void ThreadStart()
{
  using (var threadLifetime = container.BeginLifetimeScope())
  {
    var thisThreadsInstance = threadLifetime.Resolve<MyThreadScopedComponent>();
  }
}

注意：

  • 在多线程的场景下，子线程不能释放父级作用域，如果释放了，将会导致组件不能解析。

你可以在构造函数中传入一个ILifetimeScope参数把父级的生命周期作用域注入到派生线程的代码中， Autofac会自动注入当前的生命周期作用域然后你就可以从其中创建嵌套的作用域了。

public class ThreadCreator
{
  private ILifetimeScope _parentScope;
  //传入父级作用域
  public ThreadCreator(ILifetimeScope parentScope)
  {
    this._parentScope = parentScope;
  }

  public void ThreadStart()
  { //通过父级作用域创建子作用域
    using (var threadLifetime = this._parentScope.BeginLifetimeScope())
    {
      var thisThreadsInstance = threadLifetime.Resolve<MyThreadScopedComponent>();
    }
  }
}

4.3 依赖囚禁

发生条件：

一个短时间的组件被另一个长时间的组件所持有就会发生囚禁依赖。

Autofac不会阻止你创建被囚禁依赖，因为这是开发人员的责任，有时候因为囚禁的发生将会得到一个解析异常。

4.3.1 如何避免囚禁依赖

避免囚禁依赖的一般准则是：

使用该服务的组件的生命周期要小于等于正在使用的服务的生命周期。

例如：

有一个Web应用，我们通过传入请求中的一些信息来决定连接到哪一个数据库。

你可能需要以下组件：

  • 接受当前请求和数据库连接工厂的repository
  • 类似 HttpContext 的 current request ,可以用于决定业务逻辑.
  • 一个数据库连接工厂（ database connection factory ），接受一系列参数并且返回正确的数据库连接

在这个示例中，我们考虑以下应该为每个组件设定怎样的生命周期，而当前请求的上下文是一个很明显的上下文环境，我们可以将当前组件的生命周期设定为每个请求为同一实例（Instance Per Request）。

如果我们将repository设定为“单例”模式，即SingleInstance模式，那么这个实例将会缓存在应用的整个生命周期内，该请求上下文(request context)会被传入repository中，并在应用的整个生命周期中一直被repository所持有，即使当次请求已经结束了，这时候repository就是使用该服务的组件，而HttpContext就是正在使用的服务，repository是长期存在的，但是它却一直持有更小生存周期的组件，这就发生了囚禁依赖。

如果你将repository也设定为Instance Per Request模式，那么它现在和当前请求存在一样长的生命周期，它们将会被同时释放，就不存在囚禁。

再进一步，我们将repository设定为Instance Per Dependency即每一次依赖就创建一个新实例，这样也是可以的，它的生命周期比当前请求还要短，也不会发生囚禁现象。

数据库连接工厂（ database connection factory ）思考过程与之类似，但也需要根据情况分析而选择不同生命周期模式，例如实例化数据库连接工厂比较耗费资源，或者你需要维护一些内部状态它才能正常的工作，这时候可能就需要使用单例模式了。

4.3.2 应该由开发者决定是否可以接受依赖囚禁

我们在实际开发中，应该根据情况来决定是否可以接受依赖囚禁的发生。

例如你有一个缓存类（CacheClass）,它被有意设置为仅在消费组件的生命周期内缓存内容，如果该组件是单例的，那么该组件可以用于整个程序的生命周期中存储东西，如果是Instance Per Request模式，那么它只能存储单次Web请求的数据，在这种情况下选择单例模式更好，虽然发生了依赖囚禁现象，但是只要开发人员理解这样设置生存期的后果，这样也是可以接受的。

概括来说，依赖囚禁的发生应该是有意的而不是无意的。

4.4 释放

一些工作单元中包含的非托管资源例如：数据库连接、事务、文件句柄等，当工作完成时应该被释放，.net提供了IDisposable接口，让释放的概念更为明确。

在Autofac的生命周期作用域中，当工作单元完成时，嵌套的容易可以释放其中所有的实例。

using (var scope = container.BeginLifetimeScope())
{
  scope.Resolve<DisposableComponent>().DoSomething();
//using接受，该作用域中的组件都被释放
}

注意：如果注册模式为SingleInstance(),也就是单例组件，那么他们将会存在于容器的整个生命周期内，也就是应用程序的生命周期，这就意味着组件直到结束都不会被释放。，=

4.4.1 自动释放

为了实现自动释放，你的组件必须实现IDisposable接口，Disposable模式参考这篇文章。

在生命周期作用域结尾组件的Dispose()方法自动会被调用。

4.4.2 特定释放

如果你的组件没有实现IDisposable接口，但是仍然需要在生命周期作用域的结尾完成一些释放工作，你可以使用释放生命周期事件。使用 OnRelease()事件

var builder = new ContainerBuilder();
builder.RegisterType<SomeComponent>()
        //在作用域结尾调用 CleanUp方法进行释放工作
       .OnRelease(instance => instance.CleanUp());
var container = builder.Build();

4.4.3 禁止释放

我们可以禁止Autofac自动调用Dispose方法释放组件，在注册的时候使用ExternallyOwned()方法。

以这种方式注册的组件何时释放取决于你。

builder.RegisterType<SomeComponent>().ExternallyOwned();

还有一种是Owend<T>类型的自拥有组件，我们也是手动释放的。

public class Consumer
{
  private Owned<DisposableComponent> _service;

  public Consumer(Owned<DisposableComponent> service)
  {
    _service = service;
  }

  public void DoWork()
  {
    // _service is used for some task
    _service.Value.DoSomething();

    // Here _service is no longer needed, so
    // it is released
    _service.Dispose();
  }
}

4.4.4 以提供实例的方式注册

使用提供实例的方式注册，Autofac也会帮你释放该组件。

var output = new StringWriter();
//也会释放
builder.RegisterInstance(output)
       .As<TextWriter>();

4.5 生命周期事件（不太清楚作用）

4.5.1 组件被激活时的事件

激活时的事件OnActivating()在组件被使用前触发。

我们可以利用该事件：

  • 切换成另一个实例或将其包装在代理中
  • 做属性注入或者方法注入
  • 执行其它初始化任务

5 配置

可以通过JSON/XML文件配置Autofac。

5.1 使用Microsoft Configuration配置(4.0版本以上使用)

更多关于Microsoft Configuration的文档点击这里

例如一个含有简单注册的配置文件类似如下：

{
  "defaultAssembly": "Autofac.Example.Calculator",
  "components": [{
    "type": "Autofac.Example.Calculator.Addition.Add, Autofac.Example.Calculator.Addition",
    "services": [{
      "type": "Autofac.Example.Calculator.Api.IOperation"
    }],
    "injectProperties": true
  }, {
    "type": "Autofac.Example.Calculator.Division.Divide, Autofac.Example.Calculator.Division",
    "services": [{
      "type": "Autofac.Example.Calculator.Api.IOperation"
    }],
    "parameters": {
      "places": 4
    }
  }]
}

当然使用XML格式的也可以，推荐使用Json格式的，因为Json更容易读。

<?xml version="1.0" encoding="utf-8" ?>
<autofac defaultAssembly="Autofac.Example.Calculator">
    <components name="0">
        <type>Autofac.Example.Calculator.Addition.Add, Autofac.Example.Calculator.Addition</type>
        <services name="0" type="Autofac.Example.Calculator.Api.IOperation" />
        <injectProperties>true</injectProperties>
    </components>
    <components name="1">
        <type>Autofac.Example.Calculator.Division.Divide, Autofac.Example.Calculator.Division</type>
        <services name="0" type="Autofac.Example.Calculator.Api.IOperation" />
        <injectProperties>true</injectProperties>
        <parameters>
            <places>4</places>
        </parameters>
    </components>
</autofac>

之后需要让AutoFac读取配置文件：

// 添加配置文件至ConfiguraBuilder
var config = new ConfigurationBuilder();
//Microsoft.Extensions.Configuration.Json/Xml中包含config.AddJsonFile/AddXmlFile()方法
config.AddJsonFile("autofac.json");

//用Autofac注册配置模块。
var module = new ConfigurationModule(config.Build());
var builder = new ContainerBuilder();
builder.RegisterModule(module);

5.1.1 默认程序集配置

我们可以在配置文件中指定一个默认程序集的配置项，这样可以缩短写类型的事件，如果你在一个类型或者接口的引用中没有指定一个程序集的类型名称，Autofac将会默认为在默认程序集配置项中的配置。

{
  "defaultAssembly": "Autofac.Example.Calculator"
}

5.2 模块

使用配置文件这一种方式在后期不易维护，使用模块这一种方式更具有灵活性。

5.2.1 例子

模块需要实现Autofac.Core.IModule接口，通常继承Autofac.Module这一抽象类，复写Load方法，在此方法中注册组件。

代码示例：

public class CarTransportModule : Module
{
  public bool ObeySpeedLimit { get; set; }

  protected override void Load(ContainerBuilder builder)
  {
    builder.Register(c => new Car(c.Resolve<IDriver>())).As<IVehicle>();

    if (ObeySpeedLimit)
        //实例化理智司机
      builder.Register(c => new SaneDriver()).As<IDriver>();
    else
        //实例化疯狂的司机
      builder.Register(c => new CrazyDriver()).As<IDriver>();
  }
}

模块是动态的，我们可以在Load方法里做一些类似于环境检查，根据环境来注册相应的组件。

protected override void Load(ContainerBuilder builder)
{
  if (Environment.OSVersion.Platform == PlatformID.Unix)//unix系统
    RegisterUnixPathFormatter(builder);
  else
    RegisterWindowsPathFormatter(builder);//windows系统
}

从动态代理到责任链模式

代理一般来说需要两个步骤：

  • 代理对象和真实对象建立代理关系
  • 实现代理对象的代理逻辑方法

1.JDK动态代理

JDK 动态代理是 java.lang.reflect.*包提供的方式，它必须借助一个接口才能产生代理对象，

所以必须先定义一个接口：

    public interface HelloWorld {
        void sayHelloWorld();
    }

然后提供实现类HelloWordImpl来实现接口：

    public class HelloWorldImpl implements  HelloWorld{
        @Override
        public void sayHelloWorld() {
            System.out.println("Hello World");
        }
    }

以上是最简单的Java接口和实现类的关系，此时可以开始动态代理了。

就像之前所说，生成代理需要两个步骤，一是先建立起代理对象和真实服务对象的关系，然后实现代码逻辑。

在JDK动态代理中，要实现代理逻辑类必须去实现java.lang.reflect.InvocationHandler接口，它里面定义了一个invoke方法，并提供接口数组用于下挂代理对象：

    public class JdkProxyExample implements InvocationHandler {
        // 真实对象
        private  Object target = null;

        /**
         * 建立代理对象和真实对象的代理关系，并且返回代理对象
         * @param target 真实对象
         * @return 代理对象
         */
        public Object bind(Object target) {
            // 真实对象引用赋值
            this.target = target;
            // 返回代理对象
            return Proxy.newProxyInstance(
                    // 获得类加载器，此代码用的是真实对象的类加载器
                    target.getClass().getClassLoader(),
                    // 设置动态生成的代理对象挂在哪些接口下，此代码用的是真实对象的类
                    target.getClass().getInterfaces(),
                    // this代表当前对象，此对象必须实现InvocationHandler接口
                    this);
        }


        /**
         *
         * @param proxy 代理对象
         * @param method 当前调度方法
         * @param args 当前方法参数
         * @return 代理结果返回
         * @throws Throwable 异常
         */
        @Override
        public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
            System.out.println("进入代理逻辑方法");
            System.out.println("在调度真实对象之前的服务");
            Object obj = method.invoke(target,args); // 调用真实对象的方法
            System.out.println("调用真实对象之后的服务");
            return  obj; // 调用结果
        }
    }

①第一步建立代理对象和真实对象的关系

这里是用bind()方法完成的。

方法里面首先用类的属性target保存了真实对象，然后通过如下代码建立并生成了代理对象：

 Proxy.newProxyInstance(
                    // 获得类加载器，此代码用的是真实对象的类加载器
                    target.getClass().getClassLoader(),
                    // 设置动态生成的代理对象挂在哪些接口下，此代码用的是真实对象的类
                    target.getClass().getInterfaces(),
                    // this代表当前对象，此对象必须实现InvocationHandler接口
                    this);

其中newProxyInstance()方法包含了三个形参：

  • 第一个是类加载器，我们采用了target本身的类加载器。
  • 第二个是指定生成的动态代理对象的挂载接口，代码中的写法就是放在了target实现的接口下，如果真实对象是HelloWorldImpl那么它的实现接口就是HelloWorld，那么代理对象就可以这样声明：HelloWorld proxy = xxxx;
  • 第三个是定义实现方法逻辑的代理类，this代表当前对象，它必须实现InvocationHandler接口的invoke方法，它就是代理逻辑方法的实现方法。

②第二步，实现代理逻辑方法

invoke方法就是实现代理逻辑的方法，它有三个参数：

  • proxy，代理对象，就是bind方法生成的对象。
  • method，当前调度的方法。
  • args，调度方法的参数。

当代理对象调度对方法后，他就会调用invoke方法。

测试调用结果

    public void testJdkProxy() {
        JdkProxyExample jdk =new JdkProxyExample();
        // 绑定关系，bind方法中将代理对象挂载到实现类的接口下，所以可以(HelloWorld)proxy显式转换代理对象为HelloWorld
        HelloWorld proxy= (HelloWorld)jdk.bind(new HelloWorldimpl());
        // 执行代理逻辑方法invoke
        proxy.sayHelloWorld();
    }
    // 输出：
    进入代理逻辑方法
    在调度真实对象之前的服务
    Hello World 
    在调度真实对象之后的服务

2. CGLIB动态代理

因为JDK动态代理必须提供接口才能使用，在一些不能提供接口的环境中，只能采用其它第三方技术，例如CGLIB动态代理。它的优势在于不需要提供接口，只要一个非抽象类就能实现动态代理。

下面是一个例子，要给一个非抽象类ReflectServiceImpl通过CGLIB实现动态代理：

    public class ReflectServiceImpl {
        public void sayHello(String name) {
            System.out.println("hello world "+name);
        }
    }

然后编写具体的CGLIB代理逻辑：

import org.springframework.cglib.proxy.Enhancer;
import org.springframework.cglib.proxy.MethodInterceptor;
import org.springframework.cglib.proxy.MethodProxy;

import java.lang.reflect.Method;

public class CglibProxyExample  implements MethodInterceptor {
    /**
     * 生成CGLIB代理对象
     * @param cls 实际对象的Class类
     * @return Class类的CGLIB代理对象
     */
    public Object getProxy(Class cls){
        // CGLIB enhancer增强类对象
        Enhancer enhancer = new Enhancer();
        // 设置增强类型（超类）
        enhancer.setSuperclass(cls);
        // 定义代理类当前对象，要求当前对象实现MethodInterceptor方法
        enhancer.setCallback(this);
        // 生成并且返回代理对象
        return  enhancer.create();
    }


    /**
     * 代理逻辑方法
     * @param proxy 代理对象
     * @param method 方法
     * @param args 参数
     * @param methodProxy 方法代理
     * @return 代理结果返回
     * @throws Throwable
     */
    @Override
    public Object intercept(Object proxy, Method method, Object[] args, MethodProxy methodProxy) throws Throwable {
        System.out.println("调用真实对象前");
        // CGLIB反射调用真实对象方法
        Object result  = methodProxy.invokeSuper(proxy,args);
        System.out.println("调用真实对象后");
        return  result;
    }
}

这里运用了CGLIB的加强者Enhancer通过设置超类(setSuperclass)，然后通过setCallback方法设置哪个类为它的代理类。其中，参数this就意味着是当前对象，那就要用this这个对象实现接口MethodInterceptor方法—intercept，然后使用enhancer.create()返回代理对象。

此时当前类的intercept方法就是其代理逻辑方法，当调用原类的某一方法后，此方法就会被执行，我们在反射真实对象的前后进行了打印，CGLIB通过如下代码反射执行真实对象方法：

     // 执行超类的方法，也就是执行被代理对象的真实方法
    Object result  = methodProxy.invokeSuper(proxy,args);

最后测试一下代理：

        CglibProxyExample cpe = new CglibProxyExample();
        ReflectServiceImpl obj = (ReflectServiceImpl)                    cpe.getProxy(ReflectServiceImpl.class);
        obj.sayHello("wukai");

输出：

    调用真实对象前
    hello world wukai
    调用真实对象后

掌握了JDK动态代理就很容易掌握CGLIB动态代理，因为二者类似，不同的是：

  • JDK动态代理需要设置下挂接口
  • CGLIB动态代理需要设置超类

3. 拦截器

拦截器接口封装了动态代理的实现，让开发者无需知道动态代理如何实现的，只需要知道拦截器接口的方法、含义和作用即可，用JDK动态代理来实现一个拦截器的逻辑，为此先定义拦截器接口Interceptor：

public interface Interceptor {
    /** 反射方法前执行
     * @param proxy 代理对象
     * @param target 被代理真实目标对象
     * @param method 需要执行的方法
     * @param args 参数
     * @return
     */
    boolean before(Object proxy, Object target, Method method, Object[] args);

    // 实际逻辑
    void around(Object proxy, Object target, Method method, Object[] args);

    // 反射方法后执行
    void after(Object proxy, Object target, Method method, Object[] args);
}

以上代码定义了3个方法，before、around、after方法：

  • 三个方法的参数为
    1. proxy：代理对象
    2. target：被代理的真实目标对象
    3. method：执行方法
    4. args：运行参数
  • before方法的返回值是boolean，此方法在真实对象前调用，如果返回true则反射调用真实对象的方法，如果返回false，则接下来调用around方法。
  • 在反射调用真实对象方法或调用around方法之后调用after方法。

接下来用MyInterceptor实现Interceptor：

public class MyInterceptor implements  Interceptor {
    @Override
    public boolean before(Object proxy, Object target, Method method, Object[] args) {
        System.out.println("反射方法前执行逻辑");
        return false; // 返回false，不反射调用对象原有方法
    }

    @Override
    public void around(Object proxy, Object target, Method method, Object[] args) {
        System.out.println("取代被代理对象的方法");
    }

    @Override
    public void after(Object proxy, Object target, Method method, Object[] args) {
        System.out.println("反射方法后逻辑");
    }
}

MyInterceptor类实现了所有的Interceptor接口方法，接下来使用JDK动态代理，实现这些方法在适当时的调用逻辑：

import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;
import java.lang.reflect.Proxy;

public class InterceptorJdkProxy implements InvocationHandler {
    // 被代理的真实对象
    private Object target;
    // 拦截器全限定名
    private String interceptorClass = null;

    public InterceptorJdkProxy(Object target, String interceptorClass) {
        this.target = target;
        this.interceptorClass = interceptorClass;
    }

    public static Object bind(Object target, String interceptorClass) {
        // 返回被代理对象
        return Proxy.newProxyInstance(
                target.getClass().getClassLoader(),
                target.getClass().getInterfaces(),
                new InterceptorJdkProxy(target, interceptorClass));
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        if (interceptorClass == null) {
            // 没有设置拦截器，直接反射执行原对象方法
            return method.invoke(target, args);
        }
        Object result = null;
        // 通过反射生成拦截器
        Interceptor interceptor =
                (Interceptor) Class.forName(interceptorClass).getDeclaredConstructor().newInstance();
        if (interceptor.before(proxy, target, method, args)) {
            // 如果为true则反射调用原有被代理对象方法
            result = method.invoke(target, args);
        } else {
            // 若返回false则执行around方法
            interceptor.around(proxy, target, method, args);
        }
        // 最后调用后置方法
        interceptor.after(proxy, target, method, args);
        // 返回执行结果
        return result;
    }
}

这里有两个属性：

  • target：它代表被代理的真实对象
  • interceptorClass它是一个拦截器的全限定名

解释一下这段代码的执行步骤：

  • 第一步，在bind方法中用JDK动态代理绑定了一个对象，然后返回代理对象

  • 第二步，在invoke方法中编写逻辑，如果没有设置拦截器，则直接反射调用真实对象的方法，然后结束，否则进行第三步。

  • 第三步，通过反射生成拦截器

  • 第四步，调用拦截器的before方法，如果返回ture,反射原来的方法；否则运行拦截器的around方法。

  • 第五步，调用拦截器的after方法

  • 第六步，返回结果

下图是拦截器工作流程：

  • 开发者只需要知道拦截器的作用就可以编写拦截器，编写完成之后可以设置拦截器，这样就完成了任务，简化了开发流程。
  • 设计者完成动态代理的逻辑，开发者可以不用知道具体实现过程。
  • 设计者只把拦截器接口暴露给开发者使用，隐藏了动态代理的逻辑

拦截器更进一步地简化了动态代理的使用方法，使程序变得更简单：

        HelloWorld proxy = (HelloWorld) InterceptorJdkProxy.bind(
                new HelloWorldImpl(),
                "InterceptorTest.MyInterceptor");
        proxy.sayHelloWorld();

最后输出：

    反射方法前执行逻辑
    取代被代理对象的方法
    反射方法后逻辑

说明拦截器设置成功。

简而言之，使用拦截器，开发者只需要根据接口实现自己的拦截器即可，不需要关注如何实现代理类，隐藏了动态代理实现逻辑。

4. 责任链模式

现实开发中，往往需要多个拦截器来对一个流程进行拦截。

举个例子：

一个程序员需要请假一周。如果把请假申请单看成一个对象，那么它可能需要经过项目经理、部门经理、人事等多个角色的审批，没有角色都有机会通过拦截这个申请单进行审批或者修改。

把每个角色看出一个拦截器，那么就需要三个拦截器，而传递的则是请假单：

当一个对象在责任链上被多个拦截器处理（拦截器也可以选择不拦截处理它）时，我们把这样的设计模式称为责任链模式，它用于一个对象在多个角色中传递的场景。

例如请假申请这个例子，申请单走到了项目经理这，经理可能把”一周“时间改为了”5天“，从而影响了后面的审批，后面的审批都要根据前面的结果进行。这个时候可以考虑用层层代理来实现，就是当申请单（target ）走到项目经理处，使用第 个动态代理 proxyl 。当它走到部门经理处，部门经理会得到 个在项目经理的代理 proxyl 基础上生成的 proxy2 来处理经理的逻辑。当它走到人事处，会在 proxy2 基础生成 proxy3 如果还有其他角色，依此类推即可，可以用下图来描述拦截逻辑：

以之前定义的拦截器接口为例，定义三个拦截器：

拦截器1：

public class Interceptor1 implements Interceptor {
    @Override
    public boolean before(Object proxy, Object target, Method method, Object[] args) {
        System.out.println("拦截器1的 before 方法");
        // 执行被代理对象的方法
        return true;
    }
    @Override
    public void around(Object proxy, Object target, Method method, Object[] args) {
    }
    @Override
    public void after(Object proxy, Object target, Method method, Object[] args) {
        System.out.println("拦截器1的 after 方法");
    }
}

拦截器2：

public class Interceptor2 implements Interceptor {
    @Override
    public boolean before(Object proxy, Object target, Method method, Object[] args) {
        System.out.println("拦截器2的 before 方法");
        // 执行被代理对象的方法
        return true;
    }
    @Override
    public void around(Object proxy, Object target, Method method, Object[] args) {
    }
    @Override
    public void after(Object proxy, Object target, Method method, Object[] args) {
        System.out.println("拦截器2的 after 方法");
    }
}

拦截器3：

public class Interceptor3 implements Interceptor {
    @Override
    public boolean before(Object proxy, Object target, Method method, Object[] args) {
        System.out.println("拦截器3的 before 方法");
        // 执行被代理对象的方法
        return true;
    }
    @Override
    public void around(Object proxy, Object target, Method method, Object[] args) {
    }
    @Override
    public void after(Object proxy, Object target, Method method, Object[] args) {
        System.out.println("拦截器3的 after 方法");
    }
}

编写测试代码：

        // 生成原对象的代理类proxy1
        HelloWorld proxy1 = (HelloWorld) InterceptorJdkProxy.bind(
                new HelloWorldImpl(),
                "InterceptorTest.Interceptor1"); // 生成拦截器1
        // 生成proxy1的代理proxy2
        HelloWorld proxy2 = (HelloWorld) InterceptorJdkProxy.bind(
                proxy1,
                "InterceptorTest.Interceptor2"); // 生成拦截器2
        // 生成proxy2的代理proxy3
        HelloWorld proxy3 = (HelloWorld) InterceptorJdkProxy.bind(
                proxy2,
                "InterceptorTest.Interceptor3");// 生成拦截器3
        // 调用proxy3的代理方法
        proxy3.sayHelloWorld();

最后输出：

    拦截器3的 before 方法
    拦截器2的 before 方法
    拦截器1的 before 方法
    Hello World
    拦截器1的 after 方法
    拦截器2的 after 方法
    拦截器3的 after 方法

before方法执行顺序从最后一个拦截器到第一个拦截器加载顺序运行，而after方法则相反。

这时候我们如果想要多一道拦截，就可以在责任链上加入新的拦截器，增加拦截逻辑，缺点是性能不好，因为增加了代理和反射。
